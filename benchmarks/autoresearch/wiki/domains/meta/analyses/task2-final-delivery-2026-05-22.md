## 1. 范式选定

```
最终选择：黑名单（Harness 约束模式）
```

**一句话定义**：设定清晰的约束边界，在边界内 agent 可以自主决定怎么做，用户只看到输入和输出。

**核心信条**："设好围栏，围栏里你想怎么跑都行。"

---

## 2. 为什么不是白名单

白名单（Checklist）模式要求 agent 向用户逐项提问、用户逐项确认。这意味着 agent 驱动交互——**agent 问问题，用户回答**。

而 llmwiki agent 的核心使用场景正好相反：

| 场景 | 实际交互方向 | 说明 |
|------|-------------|------|
| **问答查询** | 用户 → agent | 用户提问，agent 回答。如果 agent 反问"你想要哪个数字？JCT 还是 Acc？"反而增加摩擦 |
| **Benchmark 自测** | Claude Code → agent | 测试框架输入 QA，agent 输出回答。中间不需要交互 |
| **论文 Ingest / Wiki 维护** | 用户 → agent | 用户说"帮我补充 PALCAS 的碰撞率"，agent 执行。如果 agent 逐项问"确认领域？确认格式？"用户会疯掉 |

> **结论**：llmwiki agent 的交互方向始终是 **用户→agent**，而非 agent→用户。白名单要求 agent 主动提问约束用户行为，与 llmwiki 的"你问我答"本质冲突。Harness 模式让用户自然输入，agent 负责在边界内输出正确结果——这才是正确的方向。

---

## 3. Harness 设计哲学

### 3.1 为什么适合 llmwiki

llmwiki agent 的三种核心工作，Harness 都能优雅覆盖：

| 工作类型 | Harness 下的运行方式 |
|----------|-------------------|
| **事实检索** | 用户问 → agent 读 wiki → 输出答案（附来源） |
| **综合推理** | 用户问 → agent 读多个页面 → 综合输出分析（附来源） |
| **维护修改** | 用户指令 → agent 执行修改 → 输出 diff |

三者共享同一个模式：**用户给输入，agent 在约束下自由执行，输出结果**。中间过程对用户透明。

### 3.2 与任务一的呼应

任务一中 35 题自测的结果（91.4% 完全通过率，100% 可回答率）已经验证了一个事实：**在明确的约束下，llmwiki agent 已经能可靠地完成工作**。Harness 模式要做的就是把这种"隐含约束"显式化、系统化。

## 4. 六条约束规则

### 规则 1：知识源约束 — 只读 wiki，不编造

```
约束内容
├── 所有回答必须基于 wiki 已有内容
├── 禁止编造论文细节（数字、方法、结论）
└── 每个结论必须标注来源页面（相对路径）

边界判定
├── 可回答：wiki 有直接内容 / 可从多页综合推理
└── 不可回答：wiki 无相关内容 / 需外部验证

匹配预期
├── agent 不应将"skimmed"级别的内容当作确定性事实输出
└── 跨域综合推理应标注"基于 page A + page B 综合"
```

### 规则 2：证据等级透明约束 — 你怎么知道的，告诉我

```
约束内容
├── 必须根据来源的 evidence_level 标注可靠程度
├── full-paper → "已验证" —— 定量结论可直接引用
├── skimmed → "仅读完摘要和方法" —— 核心概念可信，定量数据建议验证
└── abstract-only → "仅读摘要" —— 概览性答案

边界判定
├── 同一条回答可能混合多个等级
└── 必须明确区分"这是 wiki 上直接写的"还是"这是我综合推理的"

匹配预期
├── 用户问定量结果 → 标注对应等级
└── 用户问概念理解 → 标注来源
```

### 规则 3：诚实约束 — 不知道就说不知道

```
约束内容
├── 如果在 wiki 中找不到答案，必须明确说"找不到"
├── 不允许猜测 / 编造 / "我认为可能是"
└── 搜索失败后宣布不可回答

边界判定
├── 先检索 → 检索失败 → 不可回答
└── 可尝试从相关页面推理，但必须标注推理链

匹配预期
├── agent 应先搜索后下结论
└── "wiki 中没有记录"是合法的回答
```

### 规则 4：回答格式约束 — 结构清晰，可溯源

```
约束内容
├── 每条回答必须包含：核心结论 + 关键细节 + 来源页面
├── 推荐结构：结论 → 细节展开 → 验证页面
└── Yes/No 问答可简短，但必须有来源

边界判定
├── 单行答案必须附带来源
└── 长篇分析必须有分段和引用标记
```

### 规则 5：领域约束 — 在自家院子里活动

```
约束内容
├── 回答限定在 wiki 覆盖的 7 个领域内
└── 超出范围时礼貌告知不在覆盖范围内

覆盖领域
├── Distillation（蒸馏）
├── OOD Detection（分布外检测）
├── Spectrum（光谱分析）
├── Autonomous Driving（自动驾驶）
├── Federated Learning（联邦学习）
├── LLM Reasoning（大模型推理）
└── Meta（Wiki 结构与元分析）

边界判定
├── 边缘问题 → 在覆盖范围内尽力回答
└── 明显越界 → 告知不在覆盖范围内
```

### 规则 6：只读约束 — 问答不改内容

```
约束内容
├── 在问答模式下，agent 不得修改 wiki 文件
└── 防止"我顺手修一下"导致未审查的变更

边界判定
├── 纯问答 → 只读
├── 用户明确要求修改 → agent 输出修改建议 + 影响文件列表
└── 用户确认后 → 执行修改

匹配预期
├── agent 不应在问答中穿插写入操作
└── agent 应主动提醒"修改需要进入维护流程"
```

## 5. 约束规则的层级结构

```
六条规则按性质分三层：

🟢 安全层（不可违反）
├── 规则 1：知识源约束 —— 不编造
├── 规则 3：诚实约束 —— 不说谎
└── 规则 6：只读约束 —— 不擅自修改
    → 违反这些 = 不可接受的 agent 行为

🟡 质量层（应遵守，可商量）
├── 规则 2：证据等级透明 —— 让用户知道可信度
└── 规则 4：回答格式 —— 结构清晰
    → 违反这些 = 输出质量下降

🔵 范围层（边界界定）
└── 规则 5：领域约束 —— 在哪干活
    → 违反 = 越界，但后果相对较轻
```

## 6. 典型场景演练

### 场景 1：事实检索（最简单）

```
用户 → "FedACT 的关键数字是多少？"

agent 行为：
├── [规则 ⑤] 问题在 wiki 覆盖范围内 ✅
├── [规则 ①] 检索 fedact-concurrent-federated-intelligence.md
├── [规则 ④] 输出结构化答案
├── [规则 ②] 标注来源（full-paper 等级）
└── [规则 ③] 能找到，直接回答

输出：
"FedACT 在 4 张 A4000 GPU + i9-10900X + 64GB RAM 环境下测试。
JCT 减少 8.3×，准确率提升 44.5%。
来源：FedACT 论文页 [full-paper]"
```

### 场景 2：综合推理（高价值）

```
用户 → "TAFAP 的 trajectory alignment 和 CoRD 的 step-wise decoding
         在哲学上有什么共性？"

agent 行为：
├── [规则 ⑤] 跨域问题（distillation × LLM reasoning）✅
├── [规则 ①] 分别读取 TAFAP 页和 CoRD 页
├── [规则 ②] TAFAP full-paper ✅ | CoRD full-paper ✅
├── [规则 ①] 综合推理："过程包含快照丢失的信息"
├── [规则 ④] 结构化输出 + 标记推理链
└── [规则 ③] 可综合回答 ✅

输出：
"
核心共性：过程 > 快照
├── TAFAP：snapshot-based 保护随 fine-tuning 稀释，trajectory 控制完整优化动态
├── CoRD：post-hoc curation 丢失中间探索，step-wise 保留完整推理动态
└── 综合推理结论（基于 TAFAP 页 + CoRD 页）

来源：
- TAFAP 论文页 [full-paper]
- CoRD 论文页 [full-paper]"
```

## 7. 附录：白名单为什么不合适

### 7.1 范式本质区别

| 维度 | 白名单（Checklist） | 本选择（Harness） |
|------|:-----------------:|:----------------:|
| **交互方向** | agent → 用户（agent 提问，用户确认） | 用户 → agent（用户提问，agent 回答） |
| **适用场景** | 用户不确定该做什么，需要引导 | 用户知道自己要什么，只需要 agent 执行 |
| **典型例子** | 税务申报助手：逐项问收入/扣除项 | llmwiki：用户问"论文 X 的方法是什么？" |

### 7.2 如果 llmwiki 用白名单会怎样

```
用户想查 FedACT 的 JCT 减少
Agent 问："您好，请确认您想问的是：
  1. FedACT 论文
  2. 其他 FL 论文
  请选择 1 或 2。"
用户："1"
Agent 问："您想查什么指标？
  1. JCT 减少
  2. 准确率提升
  3. 硬件配置
  4. 以上全部"
...
```
→ 一次简单查询需要 3-5 轮交互才能拿到答案。用户明明已经在一句话里说清楚了。

### 7.3 白名单的正确使用场景

白名单模式并非不好，而是不适合 llmwiki。它适合的是：
- **用户不确定流程**的场景（如"我不懂税务，帮我报税"）
- **需要严格合规**的场景（如医疗诊断：先问症状再给建议）
- **操作不可逆**的场景（如"确认删除数据库？Y/N"）

llmwiki 的用户是**研究者**，知道自己要问什么。研究者不需要被引导，需要的是**快速、准确、可溯源的回答**。这正是 Harness 模式擅长的。

---

> **结论**：llmwiki agent 采用 **Harness（黑名单/约束边界）** 单一范式。六个约束规则覆盖安全、质量、范围三个层级，确保 agent 在围栏内自由可靠地工作。
