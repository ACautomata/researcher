## 1. Agent 概览

### llmwiki agent 是谁

一个基于 Markdown Wiki 的知识库 agent。它不依赖外部知识，所有回答都来自本地的 wiki 文件体系。

### 它的工作空间

```
D:\llmwiki\                     # 工作根目录
├── wiki\                        # wiki 内容目录
│   ├── index.md                 # 总览索引
│   ├── AGENTS.md                # agent 身份/规则定义
│   └── domains\                 # 领域目录
│       ├── distillation\
│       ├── federated-learning\
│       ├── llm-reasoning\
│       ├── outofdistributiondetection\
│       ├── spectrum\
│       ├── autonomous-driving\
│       └── meta\
├── 任务说明.docx               # 原始任务书
└── ...其他资源文件
```

### 四个核心流程

```
llmwiki agent 的四种工作模式

流程 A：问答查询 ────── 最常用，一问一答
流程 B：Benchmark 自测 ── 测试自身，输出报告
流程 C：论文 Ingest ──── 新增论文，扩充 wiki
流程 D：Wiki 维护 ────── 修复/更新已有内容
```

## 2. 全流程全景

### 2.1 抽象后的通用阶段

无论哪种流程，llmwiki agent 的工作都经过这四个阶段：

```
用户输入 ──→ ① 解析与匹配 ──→ ② 检索与读取 ──→ ③ 处理与综合 ──→ ④ 输出与交付
```

| 阶段 | 做什么 | 耗时 |
|------|--------|------|
| **① 解析与匹配** | 理解用户意图，匹配对应流程 | ~1s |
| **② 检索与读取** | 在 wiki 中找到所需文件，读取内容 | ~1-5s |
| **③ 处理与综合** | 检索 / 比较 / 推理 / 生成 / 写入 | ~5-30s |
| **④ 输出与交付** | 格式化输出 / 保存文件 | ~1s |

### 2.2 四种流程的阶段差异

| 阶段 | 流程 A（问答） | 流程 B（自测） | 流程 C（Ingest） | 流程 D（维护） |
|------|:------------:|:-------------:|:---------------:|:-------------:|
| ① 解析 | 识别问题意图 | 识别 QA pair + 判定要求 | 识别论文元数据 + 领域 | 识别维护操作类型 |
| ② 检索 | 读相关 wiki 页 | 读期望答案 + 验证页 | 读对应领域模板 + topic 页 | 读目标页面 + 交叉引用 |
| ③ 处理 | 综合推理 | 对比判定 + 统计 | 生成页面内容 + 更新索引 | 修改内容 + 更新引用 |
| ④ 输出 | 文本回答 | 测试报告 | 新 wiki 页面 | 修改后的页面 + diff |

## 3. 流程 A：问答查询

> 最核心、最常用的流程。用户问，agent 基于 wiki 回答。

### 3.1 输入

| 维度 | 说明 |
|------|------|
| **内容** | 关于 wiki 中某篇论文/某个概念的问题 |
| **格式** | 自然语言文本（中/英文均可） |
| **长度** | 不限，从一句话到一段描述 |
| **示例** | "FedACT 的关键数字是多少？" / "TAFAP 的 trajectory matching 为什么比 snapshot 好？请详细解释" |
| **约束** | 必须在 wiki 覆盖的 7 个领域内 |

### 3.2 处理流程

```
用户输入自然语言问题
    │
    ▼
① 意图识别
├── 判断是否在 wiki 覆盖范围内
├── 识别涉及的具体论文/概念
└── 识别问题类型（事实检索 / 机制理解 / 比较 / 综合推理）
    │
    ▼
② 检索策略
├── 根据问题关键词定位相关 wiki 页面
├── 读取页面内容
└── 若涉及跨域，读取多个领域的对应页面
    │
    ▼
③ 综合处理
├── 事实检索 → 提取对应段落中的结论和数字
├── 机制理解 → 理解方法逻辑，用自己的话解释
├── 跨域比较 → 分别读取后找共性和差异
└── 综合推理 → 多页信息合成新结论
    │
    ▼
④ 输出组装
├── 标注每个结论的来源页面
├── 标注证据等级（full-paper / skimmed）
└── 如果有综合推理，标注推理链
```

### 3.3 输出

| 维度 | 说明 |
|------|------|
| **内容** | 问题答案 + 来源页面 + 证据等级 |
| **格式** | 结构化文本（Markdown） |
| **长度** | 依据问题复杂度，从一句话到数百字 |
| **文件** | 不产生独立文件，直接显示为回答 |

### 3.4 示例

```
用户输入:
  "CoRD 的 predictive perplexity 比 PRM 好在哪？"

处理过程:
  ① 识别: LLM Reasoning 域，CoRD 论文，机制理解型问题
  ② 检索: 读取 cord.md 中的方法章节
  ③ 综合: PRM 需要额外训练 + 依赖最终答案正确性
           binary judgment 过粗
           predictive perplexity 在步骤粒度最优
  ④ 组装: 结论 + 细节 + 来源

输出:
  "CoRD 使用 predictive perplexity 作为步骤级选择准则。
   优于 PRM 的原因：PRM 需要额外训练且依赖最终答案正确性；
   优于 Binary Judgment 的原因：后者过粗无法区分细粒度推理质量。
   [来源：CoRD 论文页 | full-paper]"
```

## 4. 流程 B：Benchmark 自测

> 用预定义的 QA 测试集评估 agent 自身。上一轮 35 题就是这样跑的。

### 4.1 输入

| 维度 | 说明 |
|------|------|
| **内容** | 一个问题 + 期望答案 + 验证页面 |
| **格式** | QA pair（结构化文本） |
| **是否包含种子** | 每条 QA 中包含：提问文本、期望答案要点、期望答案中应出现的 source 页面 |
| **示例** | "Q: FedACT 的硬件配置？A: 4 张 A4000 GPU + i9-10900X + 64GB RAM" |
| **约束** | 期望答案必须可仅从 wiki 检索验证 |

### 4.2 处理流程

```
输入单条 QA (问题 + 期望答案 + 验证页面)
    │
    ▼
① 解析
├── 提取问题文本
└── 确认验证页面路径
    │
    ▼
② 检索
├── 读取验证页面
├── 如有需要，读相关 topic 页
└── 确认所有需要的知识都在 wiki 中
    │
    ▼
③ 回答与判定
├── 基于 wiki 内容回答问题
├── 将回答与期望答案逐点对比
└── 判定等级: ✅ 完全通过 / ⚠️ 部分通过 / ❌ 不通过
    │
    ▼
④ 记录结果
├── 记录判定结果 + 原因
├── 如果 ⚠️ / ❌，记录缺失的具体细节
└── 保存到执行记录
```

### 4.3 输出

| 维度 | 说明 |
|------|------|
| **内容** | 每条 QA 的判定结果 + 整体统计 |
| **格式** | Markdown 报告 |
| **粒度** | 逐题结果 + 汇总统计 + 改进建议 |
| **文件** | 写入 `wiki/domains/meta/analyses/` 下的报告文件 |

### 4.4 输出结构示例

```markdown
# 自测结果

## 汇总
- 总题数: 35
- ✅ 完全通过: 32 (91.4%)
- ⚠️ 部分通过: 3 (8.6%)
- ❌ 不通过: 0 (0.0%)

## 逐题详情

| 题号 | 领域 | 结果 | 说明 |
|------|------|:----:|------|
| Q5 | OOD | ⚠️ | 缺定量细节：prompt 数(8 vs 2) |
| Q9 | AD | ⚠️ | 缺碰撞率 2.45% |
| ...  |      |      |      |

## 改进建议
- P0: 补充 LSN 论文页的 prompt 数
- P0: 补充 PALCAS 论文页的碰撞率
```

## 5. 流程 C：论文 Ingest

> 将新论文加入 wiki。需要创建新页面 + 更新索引。

### 5.1 输入

| 维度 | 说明 |
|------|------|
| **内容** | 论文标题 + 作者 + 会议/期刊 + 摘要 + 方法 + 实验结果 |
| **格式** | 自然语言描述或结构化信息 |
| **来源** | 用户阅读论文后告知，或提供 arXiv 链接 |
| **示例** | "请将这篇 FedACT 论文加入 wiki，它发表在 INFOCOM 2025，主要做 multi-job FL 调度..." |
| **约束** | 必须能归属到现有的 7 个领域之一 |

### 5.2 处理流程

```
用户提供论文信息
    │
    ▼
① 信息提取与分类
├── 提取元数据（标题 / 作者 / 会议 / 年份）
├── 确定论文归属领域
├── 检查是否已存在（避免重复）
└── 判断 evidence_level（通常先标注 skimmed）
    │
    ▼
② 页面生成
├── 按领域模板创建 frontmatter（title / type / domain / status / tags）
├── 撰写摘要 / 方法 / 关键结果 / 局限性
├── 链接到已有的 topic 页 / 概念页
└── 检查与其他论文的交叉引用
    │
    ▼
③ 索引更新
├── 更新对应领域的 index.md
├── 更新总 index.md（wiki/index.md）
├── 如果涉及新概念，创建概念页
└── 检查是否需要创建 comparison 页
    │
    ▼
④ 质量检查
├── 确认所有链接是相对路径
├── 确认 topic 页索引了新论文
└── 建议 1-2 道对应的 Benchmark 测试题
```

### 5.3 输出

| 维度 | 说明 |
|------|------|
| **内容** | 新 wiki 页面 + 更新的索引页 |
| **格式** | Markdown 文件 |
| **文件位置** | `wiki/domains/{domain}/papers/{paper-file-name}.md` |
| **附加产出** | 可能同时创建/更新 concept 页、topic 页 |

### 5.4 输出结构示例

```markdown
# wiki/domains/federated-learning/papers/fedact-concurrent-federated-intelligence.md
---
title: FedACT — Concurrent Federated Intelligence
type: paper
domain: federated-learning
status: active
created: 2026-05-22
tags:
  - fl-scheduling
  - multi-job
  - fairness
source_pages:
  - wiki/domains/federated-learning/index.md
---

# FedACT

## Abstract
...

## Method
...

## Key Results
- JCT 8.3× 减少
- Accuracy +44.5%
...

## Related Pages
- [FL Heterogeneity and Optimization](../../topics/fl-heterogeneity-and-optimization.md)
```

## 6. 流程 D：Wiki 维护

> 修复/补充/更新已有 wiki 页面。来自自测发现的问题或用户主动发现。

### 6.1 输入

| 维度 | 说明 |
|------|------|
| **内容** | 维护指令：补充什么 / 修复什么 / 更新什么 |
| **格式** | 自然语言，含具体的操作描述 |
| **常见触发** | 自测报告的 ⚠️ 改进项、用户发现 wiki 缺内容 |
| **示例** | "请补充 LSN 论文页的 prompt 数和 epoch 数，之前自测发现缺了" |
| **约束** | 不能破坏已有的交叉引用 |

### 6.2 处理流程

```
用户提供维护指令
    │
    ▼
① 解析
├── 确认涉及哪些页面
├── 确认要做的操作（补充 / 修改 / 重写）
└── 评估影响范围（是否影响其他页面的引用）
    │
    ▼
② 检索现状
├── 读取目标页面的当前内容
├── 读取引用了该页面的其他页面
└── 确认需要补充的具体内容
    │
    ▼
③ 执行修改
├── 修改目标页面的对应章节
├── 更新交叉引用（如果有）
└── 补充来源标注
    │
    ▼
④ 输出 diff
├── 展示修改前后的对比
├── 列出所有被修改的文件
└── 建议是否需要相应更新 Benchmark 的期望答案
```

### 6.3 输出

| 维度 | 说明 |
|------|------|
| **内容** | 修改后的 wiki 页面 + 变更说明 |
| **格式** | 修改后的 Markdown 文件 + 文本 diff |
| **粒度** | 逐文件列出修改位置和内容 |
| **验证** | 建议修改后重新执行相关 QA 确认通过 |

### 6.4 输出结构示例

```
修改文件:
  1. LSN 论文页 +prompt 数 (8 vs 2)
  2. LSN 论文页 +epoch 数 (25 vs 5)

变更 diff:
  LSN.md:
  - [Table 5 缺具体数字]
  + [Table 5 补充: positive prompt 8个, negative 2个;
  +  positive 训练 25 epochs, negative 5 epochs]

影响范围:
  - 无交叉引用受影响
  - 建议更新 Q5 期望答案

验证:
  Q5 自测从 ⚠️ 变为 ✅
```

## 7. 输入输出总表

### 7.1 按流程汇总

| 流程 | 输入 | 输出 |
|:----:|------|------|
| **A 问答查询** | 自然语言问题 | 结构化文本答案（含来源和证据等级） |
| **B Benchmark 自测** | QA pair（问题 + 期望答案 + 验证页） | Markdown 自测报告（逐题结果 + 统计 + 建议） |
| **C 论文 Ingest** | 论文元数据 + 内容描述 | 新 wiki 页面 + 更新的索引页 |
| **D Wiki 维护** | 维护指令（补充/修复/更新） | 修改后的页面 + 变更 diff |

### 7.2 按阶段汇总

| 阶段 | 做什么 | 输入 | 输出 |
|:----:|--------|------|------|
| **① 解析** | 理解用户意图 | 用户的自然语言输入 | 结构化的操作意图（流程类型 + 目标对象 + 操作类型） |
| **② 检索** | 读取 wiki 文件 | 文件路径 / 页面名称 | 对应页面的 Markdown 内容 |
| **③ 处理** | 综合 / 对比 / 生成 / 写入 | 检索到的 wiki 内容 + 用户输入 | 答案文本 / 新页面内容 / 修改后的内容 |
| **④ 交付** | 格式化输出 | 处理结果 | 最终交付物（回答 / 报告 / 文件） |

### 7.3 文件类型总结

| 类型 | 用户给 agent | agent 给用户 |
|:----:|:-----------:|:-----------:|
| Markdown 文件 | 不多见（除非引用已有 wiki 内容） | ✅ 最常见（答案、报告、页面） |
| 自然语言文本 | ✅ 最常见（问题、指令） | ❌ 很少（除非直接对话回答） |
| 结构化数据 | ✅ Benchmark QA pair | ✅ 测试报告中的统计 |
| 纯文本 | ✅ 简单问题 | ✅ 简短回答 |
