## 任务 2：Agent 设计范式选择

**状态：✅ 已完成**

### 结论

**黑名单（Harness 约束模式）** — 设定六条约束边界，边界内 agent 自由工作，用户只看到输入和输出。

### 核心理由

llmwiki agent 的交互方向是**用户→agent**（用户提问，agent 回答）。而白名单（Checklist）要求 agent→用户（agent 逐项提问）。两者方向相反，白名单不适合"研究者知道自己要什么"的场景。

### 六条 Harness 约束规则

| 层级 | 规则 | 内容 |
|:----:|:----:|------|
| 🟢 安全层 | **R1 知识源约束** | 所有回答基于 wiki，禁止编造，标注来源页面 |
| 🟢 安全层 | **R3 诚实约束** | 找不到就说找不到，不允许猜测 |
| 🟢 安全层 | **R6 只读约束** | 问答模式不得修改 wiki 文件 |
| 🟡 质量层 | **R2 证据等级透明** | 标注 full-paper / skimmed 等级 |
| 🟡 质量层 | **R4 回答格式约束** | 结论 + 关键细节 + 来源页面的结构化输出 |
| 🔵 范围层 | **R5 领域约束** | 限定在 wiki 覆盖的 7 个领域内 |

### 交互模式

| 场景 | 用户输入 | agent 行为 |
|------|---------|-----------|
| 事实检索 | "FedACT 的关键数字？" | 检索 → 提取 → 输出结构化回答（附来源） |
| 综合推理 | "TAFAP 和 CoRD 的哲学共性？" | 多域检索 → 综合推理 → 输出（标注推理链） |
| 维护修改 | "补充 LSN 论文缺的 prompt 数" | 定位 → 修改 → 输出 diff |

## 任务 3：全流程输入输出说明

**状态：✅ 已完成**

### 四种流程全景

```
llmwiki agent 的四种工作模式：

┌─────────────────────────────────────────────────────┐
│  流程 A：问答查询 ── 最常见，一问一答                │
│  流程 B：Benchmark ── 自测，输出报告                 │
│  流程 C：Ingest ──── 新增论文，扩充 wiki             │
│  流程 D：维护 ────── 修复/更新已有内容               │
└─────────────────────────────────────────────────────┘

所有流程抽象为 4 阶段：
  用户输入 → ① 解析与匹配 → ② 检索与读取 → ③ 处理与综合 → ④ 输出与交付
```

### 各流程输入输出

| 流程 | 输入 | 处理过程 | 输出 |
|:----:|------|---------|------|
| **A 问答查询** | 自然语言问题（"FedACT 的关键数字？"） | 意图识别 → 检索 wiki → 综合推理 → 组装 | 结构化文本答案（含来源 + 证据等级） |
| **B Benchmark** | QA pair（问题 + 期望答案 + 验证页） | 解析 QA → 检索验证页 → 对比判定 → 记录 | Markdown 自测报告（逐题结果 + 统计） |
| **C Ingest** | 论文元数据 + 内容描述 | 提取分类 → 生成页面 → 更新索引 → 质量检查 | 新 wiki 页面 + 更新的索引 |
| **D 维护** | 维护指令（补充/修复/更新） | 解析范围 → 检索现状 → 执行修改 → 输出 diff | 修改后页面 + 变更 diff |

### 文件类型规范

| 类型 | 用户给 agent | agent 给用户 |
|:----:|:-----------:|:-----------:|
| Markdown 文件 | 少见 | ✅ 最常见（报告、页面） |
| 自然语言文本 | ✅ 最常见（问题、指令） | ❌ 很少 |
| 结构化数据 | ✅ Benchmark QA pair | ✅ 统计报告 |
| 纯文本 | ✅ 简单问题 | ✅ 简短回答 |
